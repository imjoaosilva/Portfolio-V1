Muito código de backend fica difícil de mudar porque a camada HTTP acaba sabendo demais. Validação, orquestração, permissões, persistência, formatação e efeitos colaterais acabam amontoados na mesma função até que nenhuma mudança pareça segura.

## Dê a cada camada um único trabalho

Um padrão mais limpo é deixar handlers traduzirem preocupações de transporte, services cuidarem do caso de uso e repositories responderem perguntas sobre dados. Isso não é cerimônia por cerimônia. É uma forma de dar a cada arquivo uma responsabilidade estreita e uma carga mental menor.

## Costuras honestas já bastam

O que mais importa é que as costuras sejam honestas. Se a camada de service só movimenta objetos sem esclarecer regra de negócio, ela não está ajudando. Mas quando o caso de uso realmente mora ali, ela vira o melhor lugar para testar e evoluir comportamento.

Esse tipo de separação também facilita consumidores futuros. Um CLI, um worker de fila, um cron ou uma segunda API podem reutilizar o mesmo service sem herdar bagagem específica da web.
